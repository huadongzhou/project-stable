# project-stable — 项目五文档基建

为项目准备稳定的根级上下文文档，让任意代码助手进入项目即拥有一致的认知基础。适用于全新项目搭建与旧项目接入 Vibe Coding 两种场合。

## 管辖范围（硬约束）

- 仅管理仓库根目录与 `templates/` 同名一一对应的五份文档：`AGENTS.md`、`PRODUCT.md`、`DESIGN.md`、`TODO.md`、`PROMPT.md`。
- 其余任何文档（README.md、CLAUDE.md、docs/ 等）不属于本技能：不创建、不修改、不迁移，仅允许作为「解读项目」时的只读信息来源。
- 用户只点名部分文档时，只处理点名的那几份；未点名操作时按「有缺失文档 → 初始化缺失的，五份齐全 → 更新」判断，并向用户说明所选操作。

## 文档定位

| 文档 | 定位 | 解读项目时的主要来源 |
|---|---|---|
| AGENTS.md | 默认上下文，最简单清晰的项目指引 | package.json、构建/lint 配置、源码目录树、README |
| PRODUCT.md | 产品指示：模块/关联/顺序/引用、增删改查关联、关系图 | 路由与菜单配置、views/modules 目录、api/types、需求文档 |
| DESIGN.md | 产品设计：设计令牌 / UX / UI 基准 | 设计稿、demo HTML、主题样式与令牌文件 |
| TODO.md | 待办：场景→功能→点 目录树 + todo 状态 | 需求输入、issue、源码 TODO 注释、git log |
| PROMPT.md | 常用提示词：BDD 场景 + 可执行提示词 | 团队工作流；通用条目开箱即用 |

## 模板机制

`templates/<名>.md` = 公共结构与公共内容（跨项目通用，原样保留） + `<!-- 生成提示：... -->` 注释（项目特性挖空点） + `{{占位符}}`。

由模板产出项目文档时：

1. 公共内容原样带入；
2. 每个生成提示 → 按提示指引解读项目，产出该处内容；
3. 产出后删除生成提示注释与未使用的占位；
4. 解读不到、无法验证的内容写「待补充：<缺什么、去哪找>」，不编造、不推定——尤其是「禁止依赖方向」这类规则性结论，无代码依据时一律待补充；
5. 某节确无内容可填时，空表头等悬空结构一并删除，只留一行待补充说明，不留残骸。

## 操作一：初始化

```gherkin
Scenario: 为项目生成缺失的上下文文档
    Given 仓库根目录缺少一份或多份模板同名文档
    When 执行初始化
    Then 逐模板执行「读取模板 → 解读项目 → 生成项目文档」，在根目录产出缺失文档
    And 已存在的同名文档保持不动，在结果中提示可改走「更新」
```

对每份待生成文档独立执行：

1. 读取 `templates/<名>.md`；
2. 按「文档定位」表列出的来源解读项目——全新项目信息不足时，以用户的需求描述为解读对象；
3. 生成 `<仓库根>/<名>.md`；
4. 汇报：已生成清单、跳过清单、各文档中的「待补充」项。

## 操作二：更新

```gherkin
Scenario: 让既有文档跟上项目现状
    Given 仓库根目录已存在模板同名文档
    When 执行更新
    Then 逐文档执行「读取模板 → 解读项目 → 更新项目文档」
    And 以最小变更落笔：补缺失章节、修正过期事实；人工补充与项目特性内容一律保留
```

更新规则：

- 模板有而文档缺的章节 → 按生成提示补齐；
- 文档有而模板没有的章节 → 保留原样（它是「同步」的输入，不是删除对象）；
- 与项目现状冲突的事实（命令、目录、依赖、令牌值）→ 核实后修正；
- 不重排既有章节顺序、不改写既有行文风格。

## 操作三：同步（反哺模板）

```gherkin
Scenario: 把项目文档中长出来的结构反哺回模板
    Given 项目文档演化出了模板中不存在的标题或内容结构
    When 执行同步
    Then 逐文档对比 <仓库根>/<名>.md 与 templates/<名>.md
    And 提取模板缺少的标题+内容结构，去除项目特性后（具体内容改写为生成提示）追加进模板
    And 模板既有内容不被项目特性值污染
```

同步规则：

- 只反哺「结构」（标题、表头、组织方式）与跨项目通用的规则；具体项目值一律改写为 `<!-- 生成提示：... -->`；
- 某结构是否具备跨项目通用性拿不准时，列出候选问用户，不默默写入；
- 方向单一：项目文档 → 模板；同步过程永不反向删改项目文档。

## 执行纪律

- 一次调用只做一种操作，初始化/更新/同步不混用；
- 所有产出基于可验证的项目事实，来源不明写「待补充」；
- 文档语言跟随项目主语言（默认中文）；
- 收尾时逐文档汇报：做了什么操作、动了哪些章节、遗留哪些待补充项。
