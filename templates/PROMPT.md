## 场景-需求完成

### 提交代码

Scenario: 将当前工作区所有变更提交到暂存区
    Given 当前工作区存在未暂存的代码变更
    When 将当前工作区的代码提交到暂存区
    Then 所有变更文件应被成功添加到暂存区
    And 工作区不应再有未暂存的修改

请审查当前 Git 暂存区（Staged）的代码变更，并严格按以下维度输出审查报告：
- 需求符合度：代码变更已与需求文档对齐（无需求变更时此项免检）。
- 破坏性影响评估：已确认不存在导致输出结果异常、接口不兼容或数据丢失的破坏性变更。
- 质量与规范达标：代码风格、注释、异常处理及安全性均符合项目开发规范。

### 修复复用

Scenario: 从暂存区变更中提取可全局复用的改进模式
    Given 当前暂存区存在已完成的代码变更
    When 归纳变更中具有通用价值的改进点
    Then 以清单呈现并经用户确认后，才对全项目做相似模式的批量修正

请分析当前 Git 暂存区（Staged）的代码变更，并按以下流程执行：
变更模式提取：从暂存区代码中识别并归纳具有通用价值的改进点，分类列出：
- 组件调整：组件引用方式变更、Props/接口适配、废弃 API 替换等；
- 代码优化：性能提升、冗余逻辑精简、可读性增强、设计模式应用等；
- 缺陷修正：Bug 修复、边界条件处理、异常捕获完善等。
交互式确认：将上述提取结果以清单形式呈现给用户，并询问："是否需要对项目进行全局扫描，查找并修正类似的模式/问题？"
按需全局治理：仅在用户明确确认后，才对全项目进行相似代码检索与批量修正；若用户拒绝，则仅保留暂存区变更，终止后续操作。

## 场景-方案生成

Scenario: 将模糊需求转化为可执行方案文档
    Given 需求只有目的与大致方向，尚无执行细节
    When 生成方案文档并放置于 agent/plan/ 中
    Then 方案应可查阅、可执行、可管理，不清楚的内容与用户交互确认

目标：给出可执行方案，放置 agent/plan/ 中，根据目的完善方案！

- 目的
- 行动

要求：可查阅/可执行/可管理

不清楚内容的可以与用户交互确认
